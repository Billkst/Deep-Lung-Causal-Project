# 项目开发经验与教训总结

## 1. 进程管理与日志监控

### 1.1 Python 输出缓冲 (Output Buffering)
- **现象**：在使用 `nohup` 或 `screen` 后台运行 Python 脚本时，日志文件（`log.txt`）长时间大小为 0，直到程序结束或缓冲区满才一次性写入。
- **原因**：Python 默认会将 stdout 缓冲。
- **解决方案**：
    1. **强制非缓冲模式**：运行命令时必须添加 `-u` 参数（如 `python -u script.py`）。
    2. **代码级刷新**：在关键 print 后调用 `sys.stdout.flush()`。
    3. **环境变量**：设置 `PYTHONUNBUFFERED=1`。

### 1.2 Conda Run 的陷阱
- **现象**：使用 `conda run -p env_path python script.py` 在后台运行时，有时会捕获 stdout/stderr 导致重定向失效或延迟。
- **原因**：`conda run` 本身是一个 wrapper，可能改变了信号传递和 IO 流的处理。
- **验证方案**：
    - **推荐**：直接使用环境内的绝对路径 Python解释器运行，例如 `/path/to/env/bin/python -u script.py`。这比 `conda run` 更直接且 IO 行为可控。

### 1.3 Screen 会话管理
- **教训**：
    - 启动前务必通过 `screen -ls` 或 `ps -ef | grep python` 检查是否有旧的僵尸进程在运行。
    - 重复启动会导致日志文件被多个进程交替写入，内容混乱。
    - 使用 `screen -S name -dm bash -c "cmd > log 2>&1"` 是标准做法，但必须确保 `cmd` 内部能够实时输出。

## 2. 实验复现性
- **必须**在脚本最开始固定所有随机种子（NumPy, PyTorch, Python random）。
- 对于消融实验或对比实验，必须使用**完全相同**的数据划分和预处理流程。


### 1.4 排查无日志的 Debug 流程
- **不要**盲目等待：如果日志文件长时间为 0 字节，不要等待超过 30 秒。
- **验证步骤**：
    1. 暂停后台任务（Ctrl+Z 或 Kill）。
    2. 在前台直接运行脚本（如 `./run.sh`）。
    3. 观察是否有控制台输出。
        - 如果无输出：说明 Python 脚本本身卡在 import 或 初始化阶段，或者 print 被禁用了。
        - 如果有输出：说明是后台工具（conda run/screen）的 IO 重定向问题。
- **案例**：2026-02-09 遇到的 ablation 实验无日志问题，最终确认为 `conda run` 吞掉 stdout 所致，改为绝对路径 python 后解决。
